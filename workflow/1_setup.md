# Workflow 1: 입력 정보 수집 (Setup)

## 목적
사용자(교사)로부터 채점에 필요한 모든 정보를 수집하고 검증한다.

## 필수 입력 7종 (v1.0)

1. **과제명** (문자열)
2. **루브릭 파일 경로** (한글·Word·PDF·이미지·텍스트 무엇이든)
3. **성취기준** (교육과정 코드 + 내용)
4. **학생 산출물 폴더 경로**
5. **채점 교사 수준**: 엄격 / 느슨
6. **의장 수준**: 엄격 / 균형 / 느슨
7. 🆕 **채점 팀 수**: 1팀 / 5팀 / 10팀 / 전체 / 사용자 지정

### 수준 선택 안내 (사용자에게 묻기)

루브릭 파싱·학생 명단 확인 후, 채점 시작 직전에 다음 세 가지를 묻는다:

```
[채점 시작 준비]

세 가지를 정해주세요.

① 채점 교사 수준 (1차 채점):
  🔵 엄격 — 글에 명백한 증거가 있어야 점수 인정
  🟢 느슨 — 학생의 시도를 적극 인정

② 의장 수준 (최종 결정):
  🔵 엄격 — 망설임 → 낮은 쪽 채택
  ⚖️ 균형 — 학생 이의·검토교사 의견 종합 (기본 권장)
  🟢 느슨 — 망설임 → 높은 쪽, 학생 시도 적극 인정

③ 채점 팀 수 (병렬 처리 단위):
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  옵션          처리 시간       토큰 비용    안정성     권장 케이스
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  1팀 (순차)    학급 30분~1시간 가장 낮음    가장 안정  적은 인원 / 점검
  5팀 (추천)    학급 5~10분    중간          안정       일반 수행평가 ⭐
  10팀          학급 3~5분     중상          안정       빠른 확인용
  전체 (=인원수) 학급 2~3분     가장 높음    조건부*     급할 때
  사용자 지정   가변            가변          가변       특정 요구
  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  * 전체 옵션은 Agent 동시 호출 한도(25) 안에서만 작동. 학생 수가 25명 초과 시 자동 분할.

추천 조합:
  · 외부 평가·중요 시험: 엄격 + 엄격 + 1팀(순차로 정밀)
  · 일반 수행평가: 엄격 + 균형 + 5팀 ⭐ (기본 권장)
  · 학기 중 성장 평가: 느슨 + 균형 + 5팀
  · 격려 중심 평가: 느슨 + 느슨 + 5팀
  · 시간이 매우 부족할 때: 어느 조합이든 + 전체/10팀
```

### 채점 팀 수 — 자세한 설명

**채점 팀이란?**
- 한 팀 = 한 학생을 처음부터 끝까지 (채점교사 → 학생이의 → 검토교사 → 의장 4단계) 처리하는 단위
- 여러 팀이 병렬로 다른 학생들을 동시에 처리

**왜 팀 수에 따라 비용·시간이 다른가?**
- **1팀**: 한 학생 끝나야 다음 학생. 총 작업량은 같지만 *순차*로 처리. 토큰 효율이 가장 좋지만 시간 길음.
- **5팀**: 5명을 한 묶음으로 동시 처리. 묶음 끝나면 다음 5명. 균형이 잡힘.
- **인원수 전체**: 모든 학생을 한 번에 동시 처리. 가장 빠름. Agent 인스턴스가 많아 토큰 소비가 살짝 높음 (각 인스턴스 컨텍스트 비용).

**왜 5팀을 추천하나?**
- 학급 25~30명 기준 약 5~10분에 완주
- Agent 동시 호출 한도(25)에 충분히 못 미쳐 안전
- 사용자가 다른 일 하는 동안 자동 완료
- 묶음 단위로 진행되어 오류 발생 시 손실 최소

사용자가 선택하면 그 조합에 해당하는 페르소나 파일을 시스템에 주입:
- 채점교사: `prompts/personas/채점교사_엄격.md` 또는 `채점교사_느슨.md`
- 의장: `prompts/personas/의장_엄격.md` / `의장_균형.md` / `의장_느슨.md`
- 처리 방식: `workflow/4_independent_judge.md`의 "팀 수에 따른 처리" 절차 적용

## 선택 입력

- **작품 텍스트** (시 감상문이라면 감상 대상 시 원문, 소설 분석이라면 작품 원문 등)
- **이전 학기 학생 글 패턴 메모** (학생을 잘 아는 교사가 작성한 사전 메모)
- **함정 패턴 주의 사항** (예: "이 평가에서는 '너=연인' 같은 편향 주의")
- **점수 산정 규칙** (충족 개수→점수 매핑이 루브릭에 없으면 추가로 받기)

## Claude 처리 절차

### 1. 입력 확인
사용자의 명령에서 4가지 필수 정보가 모두 있는지 확인한다. 빠진 정보가 있으면 한꺼번에 묻는다 (한 번씩 묻지 말 것).

```
부족한 정보 예시 질문:
"채점을 시작하려면 다음 정보가 더 필요해요.
 1. 과제명: ___
 2. 루브릭 파일: ___
 3. 성취기준: ___
 4. 학생 자료 폴더: ___"
```

### 2. 경로 유효성 검사
- 루브릭 파일이 실제로 존재하는지 `ls` 또는 Read로 확인
- 학생 자료 폴더의 파일 수·형식 확인
- 학생 자료가 여러 종(예: 시 + 해설서 + 에세이)이면 학생별로 매칭이 되는지 점검

### 3. 학생 인원 파악 + 자료 매칭
```bash
find "[학생자료폴더]" -type f | sort
```
출력에서 학번·이름 패턴을 추출해 학생 명단을 만든다. 학생당 필요한 자료 종류(시·해설서·에세이 등)가 모두 있는지 확인한다. **누락이 있으면 사용자에게 보고**한다.

### 4. 사용자에게 요약 보고
다음 형식으로 사용자에게 한 번 보여주고 진행 동의를 받는다:

```
[채점 준비 정보 확인]
· 과제명: 시 창작 및 자기 해설서
· 루브릭: examples/_레퍼런스_시창작_해설서/루브릭.png (이미지)
· 성취기준: [12문학01-03] 문학과 인접 분야의 관계를 바탕으로...
· 학생 수: 26명 (2-3반)
· 학생당 자료: 감정경험 에세이 + 창작시 + 자기 해설서 3종
· 자료 누락: 없음 (또는 "OO 학생 OO 자료 없음" 명시)

이대로 진행하시겠어요?
```

### 5. 동의 후 다음 단계로
사용자 동의를 받으면 `workflow/2_parse_rubric.md`로 진행한다.

## 주의사항

- **사용자가 4가지를 다 안 줘도 임의로 가정하지 말 것.** 누락된 정보는 반드시 묻는다.
- **학생 자료를 미리 읽지 말 것.** 이 단계는 메타 정보만 수집한다.
- 학생 명단을 만들 때 학번·이름을 정확히 추출한다. 파일명에서 추출 가능한 패턴 예시:
  - `[2-3-1] 김민준 — ...docx`
  - `[2-3-1]김민준.txt`
  - `2학년_3반_1번_김민준.hwp`
