# Workflow 4: 1라운드 독립 판정 (Independent Judgment) — v0.7

## 목적
구성된 전문가 에이전트 N명이 **서로 영향을 주지 않고 독립적으로** 각 학생을 채점한다. 이것이 토론의 출발선이 된다.

## 🆕 v0.7 핵심 변경: "1명 검증 → 모드 선택 → 자동 완주"

학급 단위 채점 시 두 가지 큰 변화:
1. **첫 학생 1명만 먼저 채점 후 사용자 검증** — 시스템이 잘 작동하는지 확인
2. **사용자가 "정석 vs 효율" 모드 선택** — 효율 모드는 묶음 처리 + 자동 완주

### 이렇게 가는 이유
학급 30명을 한 응답에 1~3명씩 끊어 진행하면 교사가 계속 응답을 기다려야 함. 교사가 **수업·회의에 다녀와도 결과가 완성**되어 있도록, 효율 모드에서는 사용자 개입 없이 끝까지 진행.

## 입력
- `outputs/[평가명]/_agent_assignments.yaml` (Workflow 3의 결과)
- 학생 산출물 (학생 자료 폴더)

## 출력
학생별로 다음 형식:
```yaml
학생_id: 2-3-1
이름: 강건우
1라운드_판정:
  - 조건_id: 1
    역할명: 시 형식 검사관
    판정: 충족  # 또는 미충족
    확신도: 98  # 0~100
    근거: "9연 27행의 분량과 수미상관 구조로 형식적 완성도 확보"
    인용: "말은 오래 머물지 못하고 / 저녁의 가장자리에 걸려있다"
    회색지대: 없음
    원_발언: |
      "시 본문을 봤습니다. 총 9연 27행으로... [실제 발언 그대로]"
  - 조건_id: 2
    ...
```

## 🆕 v0.7 학급 단위 처리 절차

학급 단위 채점(`workflow/_운용흐름.md`의 1단계) 진입 시 다음 순서로 진행:

### 단계 A: 첫 학생 1명 정밀 채점 (검증용)

1. 학생 명단에서 **첫 학생 한 명**을 선택 (학번 순)
2. 그 학생에 대해 평소 페이스로 워크플로우 4(독립 판정) ~ 5(토론) 실행
3. 결과 산출물(요약) 사용자에게 보고:

```
[첫 학생 검증 완료]

학생: [학번] [이름]
점수: [점수] / [만점]
검토 라벨: 🟢 / 🟡 / 🔴
조건별 결과:
  ① 충족 / 미충족 (확신도)
  ② ...
회색지대: [있음/없음]

이 결과가 선생님 직관과 맞는지 확인해 주세요.

다음 중 알려주시면 진행합니다:
  · "맞아, 계속해줘" → 모드 선택 단계로
  · "조건 ○이 이상해" → 그 학생만 재채점
  · "루브릭이 잘못 파싱됐어" → 루브릭 재검토
  · "잠깐 멈춰" → 작업 중단
```

### 단계 B: 사용자 검증 + 모드 선택

사용자가 "계속해줘" 또는 동의 표현을 하면, 모드 선택 안내:

```
[채점 모드 선택]

남은 학생: N명

다음 두 모드 중 선택해 주세요:

🐢 정석 모드 (Standard Mode)
  · 한 응답에 학생 1~2명씩 진행
  · 묶음마다 결과 보고
  · 선생님이 중간중간 검토하며 진행
  · 예상 응답 사이클: N/2회
  · 적합: 시간 충분, 학생마다 자세히 보고 싶을 때

🚀 효율 모드 (Efficient Mode)
  · 5명씩 묶어 자동 진행
  · 멈춤·중간 보고 X (오류 발생 시만 멈춤)
  · 모든 학생 채점 + 산출물 일괄 생성까지 자동 완주
  · 예상 시간: 30~60분 (사용자가 자리 비워도 됨)
  · 적합: 수업·회의 가야 할 때, 결과만 받아 검토하고 싶을 때

어느 모드로 가시겠어요?
```

### 단계 C-1: 정석 모드 선택 시

기존 v0.6과 동일하게 한 응답에 학생 1~2명 진행, 묶음마다 결과 보고.

### 단계 C-2: 🆕 효율 모드 선택 시 (v0.7 핵심)

다음 규칙을 따른다:

#### 효율 모드 진입 시 사용자에게 한 번 안내
```
[효율 모드 시작]

남은 학생 N명을 5명씩 묶어 자동 진행합니다.
중간에 응답을 기다리지 않습니다.
오류가 발생하면 그때만 멈춥니다.

진행 흐름:
  · 묶음 1 (학생 2~6): 채점
  · 묶음 2 (학생 7~11): 채점
  · ...
  · 마지막 묶음: 채점
  · 산출물 일괄 생성 (v0.6 차등 정책)
  · 🔴 필수검토 학생의 external_check/ 자동 생성
  · 최종 보고

선생님이 자리를 비우셔도 됩니다. 다 끝난 후 한 번에 확인해 주세요.

지금 시작합니다...
```

#### 효율 모드 행동 규칙

1. **사용자 확인 없이 다음 묶음으로 자동 진행** — 묶음 끝났다고 "다음으로 가시겠어요?" 묻지 말 것
2. **각 묶음 결과는 짧은 요약 한 줄로** (점수·라벨만), 자세한 결과는 산출물 파일에
3. **5명을 한 번에 병렬 처리** — Agent 도구를 학생당 5 페르소나 × 5명 = 최대 25개 동시 호출
4. **오류 발생 시에만 멈춤** — 학생 자료 누락, 권한 오류, 응답 형식 깨짐 등
5. **모든 학생 채점 완료 후 산출물 일괄 생성** — workflow/7_outputs.md v0.6 차등 정책 적용
6. **🔴 필수검토 학생의 external_check/ 패키지 자동 생성** — 누락 없이
7. **최종 보고는 한 번만** — 모든 작업 완료 후 종합 보고

#### 효율 모드 종료 시 보고 양식
```
[효율 모드 채점 완료]

총 처리: N명
  🟢 자동확정: X명
  🟡 권장검토: Y명
  🔴 필수검토: Z명

산출물:
  · outputs/[평가명]/채점표.md
  · outputs/[평가명]/_채점요약.md
  · 학생별 리포트·속기록 (라벨에 따라 차등)
  · 🔴 학생의 external_check/ 폴더

선생님 다음 작업:
  1. _채점요약.md 먼저 확인
  2. 🔴 학생 X명에 대해 ChatGPT/Gemini에 외부 검증
     (각 학생의 external_check/조건N_검토_요청문.md 활용)
  3. 외부 검증 완료 후 "외부 검토 완료" 알려주세요
```

## Claude 처리 절차 (단일 학생 또는 정석 모드)

### 1. 학생 자료 읽기
각 학생에 대해 모든 산출물(시·해설서·에세이 등)을 Read 도구로 읽는다. **자료를 한 번만 읽고 모든 에이전트에게 동일하게 제공**한다.

자료 변환이 필요한 경우:
- `.docx` → `textutil -convert txt -encoding UTF-8 [src] -output [dst]`
- `.hwp` → 한글 변환 도구 또는 사용자 도움 요청

### 2. 에이전트 병렬 호출

각 전문가 에이전트는 **`Agent` 도구**로 별도 호출한다. **반드시 한 메시지 안에서 N개의 `Agent` 호출을 동시에 발사**한다 (병렬 실행으로 시간 절약).

각 에이전트 호출 시 prompt 구조:

```
[페르소나]
당신은 한국 고등학교 [학년] 국어 수행평가 "[과제명]"의 
**조건 [N]만 단독 판정**하는 [역할명] 에이전트입니다.

[평가 조건 N]
"[루브릭 조건 원문]"

[평가 초점]
"[Workflow 3에서 정한 한 문장]"

[성취기준 (참고)]
"[12문학01-03] ..."

[학생 산출물]
▣ 자료 1: ...
▣ 자료 2: ...
▣ 자료 3: ...

[함정 패턴 주의 사항 - 있을 경우]
"한국어 통념상 '너=연인'으로 자동 분류하지 마세요. ..."

[지시]
오직 조건 [N]만 판정합니다. 다른 조건은 보지 마세요.

다음 형식으로 한국어로 보고하세요 (300자 이내):

판정: 충족(O) 또는 미충족(X)
확신도: 0~100 (%) — 회색지대일수록 50에 가깝게
근거: (2~4줄. 학생 자료의 어디서 무엇을 봤는지 구체적으로)
인용: (학생 작품/해설서에서 직접 인용 1~2구절. 페이지·줄 단위로 정확히)
회색지대: 있음/없음 — 있으면 다른 채점자가 이견을 가질 만한 지점 한 줄
신뢰도: 높음/중간/낮음
원_발언: |
  ["내가 생각하기로는..." 식의 1인칭 회의 발언체. 토론 속기록에 그대로 들어감.
   200~400자. 자신의 사고 과정을 그대로 보여주듯이.]
```

### 3. 객관형 vs 해석형 에이전트의 차이

| 차원 | 객관형 에이전트 | 해석형 에이전트 |
|---|---|---|
| 추가 지시 | "이진 판정에 가깝게. 확신도 80% 이상이 자연스럽다." | "다층 해석을 검토하라. 회색지대를 적극적으로 발견하라." |
| 인용 요구 | 텍스트 1구절 | 본문·해설서·에세이를 가로지르는 매핑 |
| 신뢰도 보고 | 보통 "높음" | "높음~낮음" 분포 자연스러움 |

### 4. 응답 수집·검증
모든 에이전트가 응답하면:
- 응답 형식이 표준에 맞는지 검사
- 형식 오류가 있으면 해당 에이전트만 재호출 (전체 재호출 X)
- `원_발언` 필드가 비어 있으면 반드시 채워서 받기 (속기록에 필요)

### 5. 결과 저장
학생별 1라운드 판정을 `outputs/[평가명]/per_student/[학번]_이름/01_round1.yaml`에 저장.

### 6. 다음 단계 판단
1라운드 결과를 보고:
- **모든 항목이 객관형이고 모두 확신도 80% 이상** → Workflow 5(토론) 건너뛰고 바로 Workflow 6(의장)로 진행
- **해석형 항목이 있거나 회색지대·미충족이 있음** → Workflow 5로 진행

## 주의사항

- **에이전트 간 정보 공유 금지.** 각 에이전트는 다른 에이전트의 판정을 알지 못해야 한다 (동조 방지).
- **자료를 임의로 요약하지 말 것.** 각 에이전트는 학생 자료 전문을 받아야 한다.
- **`원_발언`은 진짜 사람이 회의에서 말하듯이.** "판정: 충족 / 근거: ..." 같은 표 형식이 아니라 "제가 시 본문을 봤는데요..." 식의 1인칭 발언.
- **확신도는 신뢰도와 다르다.** 확신도는 판정의 강도, 신뢰도는 자기 판정에 대한 자기 평가. 둘 다 받기.
- 학생 산출물에 위조·모순 정황이 있으면 반드시 명시하라 (예: 학생이 인용한 에세이 문장이 실제 에세이에 없음).
